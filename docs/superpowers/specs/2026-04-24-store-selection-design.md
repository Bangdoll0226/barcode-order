# 店舗選択機能 設計書

- 作成日: 2026-04-24
- 親プロジェクト: `barcode-order`（既存アプリへの機能追加）
- 親仕様: `2026-04-24-barcode-order-design.md`

## 1. 概要

複数店舗を1つのスマホで使い回す状況に対応するため、「店舗名」を選択できるようにする。
- 初回起動時に店舗を選択（強制）
- 選択した店舗は localStorage に保存され、以降ずっとその店舗で使われる
- メール件名・本文・CSV ファイル名すべてに店舗名が反映され、受信側でどの店舗からの発注か瞬時に判別できる
- 店舗マスタは管理画面で追加・編集・削除・並び替え可能（発注先マスタと同じパターン）

## 2. 要件

### 機能要件

1. 店舗マスタ（複数店舗の登録・管理）
2. 起動時の店舗選択（current_store が未設定なら強制モーダル）
3. ヘッダーに現在の店舗名を表示し、タップで切替えできる
4. メール件名: `[店舗名] 発注リスト YYYY-MM-DD HH:mm`
5. メール本文冒頭: `店舗名 からの発注リストをお送りします。`
6. CSV ファイル名: `order-{店舗名}-{発注先名}-YYYYMMDD-HHmm.csv`
7. 現在選択中の店舗は店舗マスタから削除不可（誤操作防止）
8. 店舗が1件しかない場合、その店舗は削除不可
9. 発注リスト中に店舗を切替えても、リストは消さない（同じスマホで店舗をまたがる作業を許可）

### 非機能要件

- 既存のデータ（商品マスタ・発注先マスタ・発注リスト）への影響なし
- localStorage のスキーマ変更は追加のみ（既存キーは変更しない）

## 3. データモデル

### 3.1 店舗マスタ（localStorage キー: `barcode-order:stores`）

```json
[
  { "name": "A店" },
  { "name": "B店" }
]
```

- 配列。並び順が UI 表示順
- `name`: 必須・重複不可
- 発注先マスタと同じ操作セット（add/update/remove/move）

### 3.2 現在選択中の店舗（localStorage キー: `barcode-order:current_store`）

```json
"A店"
```

- 単純な文字列（JSON エンコード）
- null/未設定なら「初回起動」とみなして店舗選択モーダルを表示

## 4. UI 変更

### 4.1 ヘッダー

現在のヘッダーに店舗表示を追加：

```
┌──────────────────────────────────────┐
│  バーコード発注                ログイン │
│  🏪 A店                                │
└──────────────────────────────────────┘
```

- `🏪 A店` の部分はクリッカブル → 店舗切替モーダル
- 未設定状態では `🏪 店舗未設定` と表示（強制モーダルで通常は到達しない）

### 4.2 起動時フロー

```
init()
  ↓
current_store が設定済み？
  ├── Yes → そのまま render
  └── No  → 強制店舗選択モーダル
            ├── 店舗マスタが空 → 「最初の店舗名を入力してください」入力欄
            │                   入力 → addStore + setCurrentStore → モーダル閉じる
            └── 店舗マスタにあり → ラジオボタン一覧
                                   選択 → setCurrentStore → モーダル閉じる
```

- このモーダルは × ボタンや背景タップで閉じない（必須選択）

### 4.3 店舗切替モーダル

ヘッダーの店舗名タップで開く：
- 現在の店舗一覧（ラジオボタン）
- 「店舗マスタを管理する」ボタン（タップで店舗マスタ管理セクションへ）
- 「キャンセル」ボタン

切替時、発注リストに行があれば確認ダイアログ：
> 店舗を変更すると現在の発注リストはそのまま残ります。よろしいですか？

### 4.4 操作エリアにボタン追加

現在の4ボタンに「店舗マスタを管理」を追加。発注先マスタの隣に配置。

### 4.5 店舗マスタ管理セクション

発注先マスタと同じ UI パターン：
- 店舗一覧（編集可能なテキストフィールド）
- 上下ボタンで並び替え
- 削除ボタン（使用中チェック付き）
- 末尾に「追加」フォーム

## 5. 出力フォーマット

### 5.1 メール件名

```
[A店] 発注リスト 2026-04-24 22:30
```

`buildSubject(date, storeName)` で生成。

### 5.2 メール本文（HTML）

冒頭に1行追加：

```html
<p>A店 からの発注リストをお送りします。</p>
```

その後は既存のフォーマット（発注先ごとのアイテム + リンク）。

### 5.3 CSV ファイル名

```
order-A店-アスクル-20260424-2230.csv
```

`generateFilename(storeName, supplierName, date)` に変更（store 引数を先頭に追加）。

### 5.4 CSV 中身

**変更なし**。`品名,数量,JANコード` の3列のまま。
- 店舗情報はファイル名・件名・本文で十分
- 既存ロジックを温存

## 6. 主要関数

### storage.js（追加）

- `getStores()`: 店舗マスタ取得（空配列を返す。デフォルト投入なし）
- `addStore({ name })`: 追加（重複チェック）
- `updateStore(oldName, { name })`: 名称変更
- `removeStore(name)`: 削除
- `moveStore(name, delta)`: 並び替え
- `getCurrentStore()`: 現在選択中の店舗名（未設定なら null）
- `setCurrentStore(name)`: 現在選択中の店舗を設定

### app.js（変更・追加）

- `renderStoreHeader()`: ヘッダーの店舗名表示更新
- `openStoreSelectionModal({ required })`: 店舗選択モーダル（required: true なら閉じれない）
- `openStoreSwitchModal()`: 店舗切替モーダル（既存リスト警告つき）
- `renderStoreMaster()` / `toggleStoreMaster()`: 店舗マスタ管理 UI
- `init()` を変更: 店舗未設定なら `openStoreSelectionModal({ required: true })`
- `handleSendEmail()` を変更: 件名・本文・ファイル名に store 名を反映

### csv.js（変更）

- `generateFilename(store, supplier, date)`: 引数の並びを変更（store を先頭追加）

## 7. エラーハンドリング

| ケース | 対応 |
| --- | --- |
| current_store が空でメール送信ボタンを押した | 通常起きない（強制モーダルで防止）。万が一は「店舗が選択されていません」エラー |
| 店舗マスタの削除で current_store と一致 | 「現在使用中の店舗のため削除できません。先に他の店舗に切替えてください」 |
| 店舗が1件しかない状態でその1件を削除 | 上記と同じ理由で削除不可（current_store と一致するはず） |
| 店舗マスタで重複名を追加 | 「この店舗名は既に存在します」 |
| 店舗名を空文字で追加・編集 | 「店舗名は必須です」 |

## 8. テスト方針

### 自動テスト（test.html に追加）

- 店舗マスタ CRUD（getStores, addStore, updateStore, removeStore, moveStore）
- 重複追加で例外
- getCurrentStore / setCurrentStore の往復
- `csv.generateFilename(store, supplier, date)` の新シグネチャ

### 手動テスト（実機）

1. localStorage を完全クリアした状態でアプリを開く → 強制モーダルが出る
2. 店舗名を入れて確定 → ヘッダーに店舗名が表示される
3. 発注リストに数行入れてメール送信 → 件名・本文・添付ファイル名すべてに店舗名が入っている
4. 店舗マスタで2店舗目を追加
5. ヘッダーの店舗名タップ → 切替モーダルが出る → 別店舗を選択
6. 発注リストに行があるとき切替 → 確認ダイアログが出る
7. 現在選択中の店舗を削除しようとする → ブロックされる
8. 店舗マスタから他店舗に切替 → 元の店舗が削除可能になる

## 9. スコープ外

- 店舗ごとに別の発注リストを持つ（マルチプロジェクト的な使い方）
- 店舗ごとの商品マスタ分離（現状すべて共通マスタ）
- 店舗別の発注先デフォルト値
- 店舗ごとの送信先メールアドレスの分離

これらは将来的に必要になれば別仕様として検討する。
