# 発注履歴機能 設計書

- 作成日: 2026-04-24
- 親プロジェクト: `barcode-order`（既存アプリへの機能追加）
- 親仕様: `2026-04-24-barcode-order-design.md`、`2026-04-24-store-selection-design.md`

## 1. 概要

メール送信した発注リストを自動的に履歴として保存し、アプリ上で過去の発注内容を閲覧・復元できるようにする。
- 送信時に自動保存（ユーザー操作不要）
- 店舗ごとに最大10件保持（古いものは自動削除）
- 現在選択中の店舗の履歴のみを一覧表示
- 過去の発注リストを「上書き」または「追加」で現在の発注リストに復元できる
- 月次・週次の定期発注を効率化

## 2. 要件

### 機能要件

1. メール送信成功時に、送信した発注リスト全体を履歴に自動保存
2. 履歴は店舗ごとに最大10件（11件目を保存しようとしたら最古を削除）
3. 操作エリアの「発注履歴を見る」ボタンで履歴一覧モーダルを開く
4. 履歴一覧は現在選択中の店舗のもののみ、新しい順
5. 各履歴エントリで「詳細」ボタンを押すと品目一覧を含む詳細モーダルが開く
6. 詳細モーダル内の「このリストを復元」ボタンで、上書き / 追加 / キャンセルの3択ダイアログ
7. 「上書き」: 現在の発注リストを削除して履歴の内容で置き換え
8. 「追加」: 履歴の各行を現在のリストにマージ（同じ JAN は数量合算）

### 非機能要件

- 履歴の保存失敗（localStorage 満杯など）はメール送信機能を止めない
- 既存の店舗・商品・発注先マスタへの影響なし

## 3. データモデル

### 3.1 履歴（localStorage キー: `barcode-order:history`）

```json
[
  {
    "id": "uuid-x",
    "sentAt": "2026-04-24T22:30:00+09:00",
    "store": "A店",
    "lines": [
      { "barcode": "4901234567890", "name": "アルコール綿 100包", "quantity": 3, "supplier": "アスクル" },
      { "barcode": "4909876543210", "name": "ディスポ手袋 M", "quantity": 5, "supplier": "シンリョウ" }
    ]
  }
]
```

- 配列。**全店舗のエントリを混在で保持**（フィルタは取得時に行う）
- `id`: `crypto.randomUUID()` または既存の `newId()` ヘルパー
- `sentAt`: `new Date().toISOString()`
- `store`: 送信時の店舗名
- `lines`: 送信時の発注リストの配列（id と addedAt は省略する。barcode/name/quantity/supplier のみ）
- 上限: 店舗ごとに10件（追加時に該当店舗の件数をチェックし、11件以上なら最古を削除）

## 4. 主要ロジック

### storage.js（追加）

- `addHistoryEntry({ store, lines })`:
  1. 全履歴を取得
  2. `{ id: newId(), sentAt: new Date().toISOString(), store, lines }` を末尾に追加
  3. 同じ `store` のエントリを抽出 → 古い順に並べ → 11件以上なら最古を削除
  4. 履歴を保存
- `getHistory(storeName)`:
  - 全履歴から `entry.store === storeName` をフィルタ
  - `sentAt` の降順（新しい順）でソートして返す

### app.js（追加・変更）

- `handleSendEmail()` を変更: 送信成功後、`storage.clearOrder()` の**直前**で履歴保存を呼ぶ。失敗してもメール送信エラーにはせず、トーストのみ表示
- `openHistoryListModal()`: 履歴一覧モーダルを開く
- `openHistoryDetailModal(entry)`: 詳細モーダルを開く
- `restoreHistoryEntry(entry, mode)`: `mode = "overwrite" | "append"` で復元
- `wireActions()` に「発注履歴を見る」ボタン listener を追加

### 5.4 復元方法選択ダイアログ

```
┌────────────────────────────────────────┐
│ どう復元しますか？                     │
│                                        │
│ ・上書き: 現在のリストを破棄して        │
```

- 「上書き」と「追加」は別々のボタン（タップで即実行 + 結果トースト）
- 実行後、履歴一覧モーダルも詳細モーダルも閉じる
- 発注リストを再描画

## 6. 復元ロジック

### 上書き（overwrite）

1. `storage.clearOrder()` を呼ぶ
2. 履歴の `lines` 配列を順に処理：
   - 各 line に対して `storage.addOrderLine({ barcode, name, supplier })` を呼ぶ（quantity は1で初期化される）
   - その後 `quantity - 1` 回 `storage.incrementQuantity(barcode)` を呼んで指定数量に揃える
3. `renderOrderList()` を呼ぶ
4. トースト: `履歴を復元しました（{N}件）`

### 追加（append）

1. 履歴の `lines` 配列を順に処理：
   - 現在の発注リストに同じ `barcode` の行があれば、`quantity` 回 `storage.incrementQuantity(barcode)` を呼ぶ（合算）
   - なければ `storage.addOrderLine({ barcode, name, supplier })` で新規追加 → その後 `quantity - 1` 回 `incrementQuantity` で指定数量に
2. `renderOrderList()` を呼ぶ
3. トースト: `履歴を追加しました（{N}件）`

## 7. エラーハンドリング

| ケース | 対応 |
| --- | --- |
| 送信成功後の履歴保存で例外（localStorage 満杯等） | catch して `showToast("履歴の保存に失敗しました")` 表示。メール送信全体は成功扱い、発注リストはクリアする |
| 履歴一覧で履歴ゼロ件 | 「まだ履歴がありません」 |
| 履歴の発注先が現在の発注先マスタに無い | 復元するとそのまま supplier 名で行が入る。既存の「削除済」フォールバックUIで表示される |
| 履歴の商品が現在の商品マスタに無い | 復元時に supplier/name/quantity を持つ行として追加。商品マスタへは登録しない（過去発注の復元のみ） |
| 「上書き」を選んだとき現在の発注リストが空 | 確認なしで実行（3択ダイアログのまま、特別な分岐なし） |

## 8. テスト方針

### 自動テスト（test.html に追加）

- `addHistoryEntry`: 1件追加 → `getHistory(store)` で取得
- `addHistoryEntry`: 11件追加 → 該当店舗のエントリが10件で、最古が消えていることを確認
- `addHistoryEntry`: 別店舗のエントリは10件制限の対象外（A店11件目入れても B店3件は残る）
- `getHistory`: フィルタが店舗名で正しく動く
- `getHistory`: 新しい順にソート（sentAt 降順）

### 手動テスト

1. 発注リストに数件入れて送信 → 履歴一覧モーダルに1件出る
2. 「詳細」をタップ → 発注先別にグルーピングされた品目が見える
3. 「このリストを復元」→「上書き」→ 発注リストが置き換わる
4. 「このリストを復元」→「追加」→ 同じ商品なら数量が増える
5. 別店舗に切替 → 履歴一覧が空または別店舗の履歴に切り替わる
6. 11回送信 → 最古が消える（直近10件のみ表示）

## 9. スコープ外

- 履歴の手動削除（10件で自動回転するため不要）
- 履歴のエクスポート（CSV/JSON）
- 履歴の検索・絞り込み（10件しかないので不要）
- 履歴からの再送信（シンプルさ優先で復元のみ）
- 全店舗横断の履歴ビュー
