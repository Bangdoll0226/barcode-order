# 発注カート自動投入 設計

barcode-order アプリが Gmail に送信した発注 CSV を起点に、Claude が Chrome を操作して各発注先サイトのカートに商品を投入する仕組みの設計。決済ボタンはユーザーが手動で押す。

## 目的

barcode-order アプリの出力 CSV を「発注先サイトでのカート投入」まで自動化し、ユーザーは最終確認と決済だけを行えばよい状態にする。

## 全体フロー

1. ユーザーが Claude に `/order` コマンド、または「発注やって」「カート投入して」等の自然言語で起動を依頼。
2. Claude が Gmail を検索して最新の発注メールを取得（barcode-order が自分宛に送信したもの）。
3. メールに添付された複数の CSV を取得・パース。ファイル名 `order-{店舗}-{発注先}-YYYYMMDD-HHMM.csv` から発注先名を抽出する。
4. 発注先名を `barcode-order/supplier-sites.json` のマッピングで URL に変換。未知の発注先は都度ユーザーに URL を確認し、JSON に追記。
5. 発注先ごとに新規タブで対象サイトを開き、ログイン状態を確認。
6. CSV の各行について、JAN コード検索 → 品名フォールバック → 判別不能ならスキップ、の順に商品を特定してカートに数量分投入。
7. 1 発注先の処理が終わったらカート画面で停止し、次の発注先の新規タブを開いて続行。
8. 全発注先の処理が完了したら、Claude が最終レポート（投入成功一覧 + 手動対応が必要な行の一覧）をターミナルに表示。
9. ユーザーが各タブのカート画面を確認し、決済ボタンを押す（Claude は決済操作を行わない）。

## 起動方法

- 主トリガー: スラッシュコマンド `/order` を skill として定義し、起動を明示できるようにする。
- 副トリガー: `/order` skill の description に「発注やって」「カート投入して」等の自然言語フレーズを記載し、これらが含まれる発話でも skill が発火するようにする。

両方を共存させることで、明示性と気軽さを両立する。

## 入力

### Gmail メールの取得条件

- 送信者: ユーザー自身（`from:me`）
- 添付有無: あり（`has:attachment`）
- 件名キーワード: `発注リスト`（barcode-order の `buildSubject` が `[{店舗名}] 発注リスト YYYY-MM-DD HH:MM` という件名を生成するため）
- 取得対象: 上記条件にマッチする最新 1 件

該当メールが見つからない場合は、Claude が「最新の発注メールが見つからない。期間や件名を絞れる？」とユーザーに確認する。

### CSV の構造

- ヘッダ: `品名, 数量, JANコード`
- BOM 付き UTF-8 / CRLF 改行
- 各行が 1 商品

### CSV ファイル名

- 形式: `order-{店舗}-{発注先}-YYYYMMDD-HHMM.csv`
- 発注先名は 3 番目のセグメントから抽出する。

## 発注先マッピング

### 保管場所

`barcode-order/supplier-sites.json`

### 形式

```json
{
  "アスクル": "https://www.askul.co.jp/",
  "カウネット": "https://www.kaunet.com/",
  "Amazon": "https://www.amazon.co.jp/",
  "シンリョウ": "https://www.shinryo.jp/"
}
```

### 未知発注先の追加フロー

CSV から取り出した発注先名がこの JSON にない場合:

1. Claude がユーザーに「『◯◯』のサイト URL を教えて」と確認。
2. ユーザーから受け取った URL を JSON に追記。
3. 次回以降は確認なしで使う。

### barcode-order 側マスタとの関係

barcode-order アプリ側にも発注先マスタ（`name`, `url`）が localStorage 内にあるが、これは Claude から直接読めない。`supplier-sites.json` は barcode-order 側マスタの「Claude 用ミラー」として独立に管理し、ユーザーが手動で同期する（食い違っても致命的ではないが、URL を変更したら両方更新するのが望ましい）。

## ブラウザ操作

### 操作手段

`mcp__Claude_in_Chrome__*` ツール（Claude for Chrome MCP）を使い、ユーザーが普段使っている Chrome に接続して操作する。これにより既存のログイン Cookie / セッションがそのまま使える。

### タブ管理

- 発注先 1 つにつき新規タブ 1 つ。
- 1 タブ内ですべての検索・カート投入を行う（タブ切り替え不要）。
- 1 発注先の処理完了時点でそのタブはカート画面に遷移して停止。
- 次の発注先は別の新規タブで開始。
- 全発注先処理後、すべてのタブがそれぞれカート画面で開いた状態のまま残す。

### ログイン状態の確認

各タブを開いた直後に、サイトがログイン済みかをチェック（ヘッダのアカウント名表示等で判定）。未ログインの場合:

1. その発注先の処理を停止。
2. ユーザーに「{サイト} が未ログイン状態。Chrome 上でログインしてから『続けて』と言って」と通知。
3. ユーザーが手動でログインして「続けて」と発話したら、その発注先から処理を再開。

## 商品マッチングロジック

各 CSV 行（`{品名, 数量, JANコード}`）について、以下の段階で商品を特定する。

### 第1段階: JAN コード検索

- サイトの検索ボックスに 13 桁の JAN コードを入力して検索。
- ヒット 1 件のみ → その商品を採用。
- ヒット複数件 → 第3段階。
- ヒット 0 件 → 第2段階。

### 第2段階: 品名検索（フォールバック）

- サイトの検索ボックスに CSV の `品名` を入力して検索。
- 検索結果の中に JAN コードが完全一致する商品があれば、それを採用。
- JAN 一致がない場合、品名が CSV の `品名` と完全一致する商品があれば採用。
- それ以外は第3段階。

### 第3段階: 判別不能 → スキップ

- カート投入は行わない。
- 最終レポートに記録（理由: ヒット無し / 複数候補で判別不能 等）。
- 次の行に進む。

### 採用後の処理

1. 商品ページに遷移、または検索結果から数量入力可能な箇所に移動。
2. 数量を CSV の `数量` 列の値に設定。
3. 「カートに追加」相当のボタンをクリック。
4. カート追加が成功したことを確認（成功メッセージ、カート件数更新等）。
5. 確認できたら次の行へ。失敗したら第3段階と同じくスキップしてレポート記録。

### 安全装置

- 「複数候補のうち先頭を選ぶ」「価格や名称の近さで自動判定する」といった**曖昧な自動選択は行わない**。誤発注を避けるため、判別不能は必ずスキップする。
- 在庫切れ・販売終了の商品は「ヒット 0 件」と同等に扱う。

## 進捗表示

Claude は処理中、ターミナルに以下のような形式で逐次進捗を表示する。

```
[1/3] Amazon の処理開始 (5件)
  ✓ 「コクヨ ノート キャンパス A4」 (JAN 4901234567890) → カート投入 (数量1)
  ✓ 「ぺんてる ボールペン 0.7mm」 (JAN 4902345678901) → カート投入 (数量2)
  ✗ 「ぺんてる ボールペン 0.5mm」 (JAN 4903456789012) → ヒット無し、品名でも見つからずスキップ
  ✓ ...
  Amazon 完了: 4/5 件投入

[2/3] シンリョウ の処理開始 (3件)
  ...
```

## 最終レポート

全発注先処理後、以下の形式で表示する。

```
=== 発注処理完了 ===

✅ カート投入完了:
  - Amazon: 4件
  - シンリョウ: 3件
  - アスクル: 6件

⚠️ 手動対応が必要:
  - Amazon: 「ぺんてる ボールペン 0.5mm」 (JAN 4903456789012) — ヒット無し
  - シンリョウ: 「アルコール綿 100枚」 (JAN xxxxxx) — 複数候補で判別不能（容量違い等が混在）

各タブのカート画面が開いています。内容を確認して決済ボタンを押してください。
```

商品名・JAN コード・スキップ理由を必ず含め、ユーザーが手動対応する際にすぐ商品を特定できるようにする。

## エラーハンドリング

| 状況 | 対応 |
|---|---|
| Gmail に該当メールが見つからない | 処理を停止し、件名・期間の絞り込みをユーザーに確認 |
| サイトが未ログイン状態 | その発注先の処理を一時停止、ユーザーに通知、「続けて」で再開 |
| ページ構造が想定と違ってカート投入できない | その行はスキップしてレポート記録、次の行へ |
| サイト全体が重い・落ちている | その発注先 1 つだけスキップ、レポート記録、次の発注先へ |
| Cookie 同意・通知許可等の明らかなポップアップ | Claude が判断して閉じる。判断つかないものは停止して確認 |
| 1 件の処理が 1 分以上かかる | タイムアウトとしてスキップ、レポート記録 |
| Claude for Chrome 自体が応答しない | 処理を停止、拡張機能の状態確認をユーザーに依頼 |

「続けて」コマンド: 一時停止後にユーザーが「続けて」と発話したら、停止地点から処理を再開する。

## 動作確認

スクリプトではなく Claude が直接操作するため、ユニットテストは行わない。代わりに初回稼働時に以下を確認する。

1. **Gmail MCP の疎通**: Claude が最新発注メールを取得できる。
2. **Claude for Chrome の疎通**: Claude が Chrome で新規タブを開き、ナビゲートできる。
3. **各発注先サイトのログイン状態**: ユーザーが先に Chrome で各サイトにログインしている。
4. **小規模ドライラン**: 1 発注先・3〜5 件の小さいバッチで一通り通す。最終レポートが想定通りであることを確認。

問題がなければ通常運用に移行する。

## YAGNI とした項目

将来の拡張として議論したが、今回のスコープからは外したもの:

- **サイト別の固定レシピ（DOM セレクタ等のキャッシュ）**: 案1の方針として、Claude が毎回その場でサイトを見て判断する。固定レシピを書くと安定するが、サイト変更時のメンテが必要になり、メンテフリーという案1の利点を損なう。必要になれば後から導入する。
- **定期実行 / Gmail 着信時自動起動**: 手動トリガーで運用してから判断する。
- **複数の発注メールをまたいだ処理**: 今は最新 1 通のみ対象。
- **発注後の在庫管理連携**: スコープ外。

## 用語

- **発注先**: barcode-order アプリで商品を割り当てる供給元（アスクル、Amazon、シンリョウなど）。1 つの発注メールに複数の発注先 CSV が添付される。
- **店舗**: barcode-order の `current_store`（複数店舗を持つユーザー向けの区分）。CSV ファイル名に含まれる。
- **発注メール**: barcode-order アプリが Gmail API 経由で自分宛に送信する、複数 CSV 添付のメール。
